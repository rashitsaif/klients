# 03_ARCHITECTURE_RULES.md

## Назначение

Этот файл задает архитектурные правила для стабильных функциональных приложений.

Главная цель архитектуры — чтобы приложение можно было:

- запустить;
- понять;
- проверить;
- доработать;
- масштабировать;
- безопасно выпускать.

---

## Стек по умолчанию

Если пользователь не указал другой стек, использовать:

Frontend:

- React;
- TypeScript;
- Vite;
- Tailwind CSS;
- shadcn/ui при необходимости.

Backend:

- Supabase Auth;
- PostgreSQL;
- Row Level Security;
- Supabase Edge Functions.

AI API:

- только через backend.

Платежи:

- только через backend + webhook.

---

## Базовая структура frontend

Рекомендуемая структура:

```text
src/
  app/
  pages/
  components/
  features/
  hooks/
  lib/
  services/
  types/
  constants/
  styles/
```

Назначение:

- `app/` — инициализация приложения, провайдеры, роутинг.
- `pages/` — страницы маршрутов.
- `components/` — переиспользуемые UI-компоненты.
- `features/` — функциональные модули.
- `hooks/` — пользовательские хуки.
- `lib/` — вспомогательные библиотеки и клиенты.
- `services/` — API-запросы и внешние взаимодействия.
- `types/` — общие типы.
- `constants/` — константы.
- `styles/` — стили.

---

## Разделение ответственности

Запрещено складывать всю логику в один компонент.

Разделять:

- UI;
- бизнес-логику;
- API-запросы;
- типы;
- валидацию;
- работу с состоянием;
- форматирование данных.

Компонент страницы не должен превращаться в огромный файл с полной бизнес-логикой.

---

## Правила компонентов

Компоненты должны быть:

- понятными;
- небольшими;
- типизированными;
- переиспользуемыми, если это оправдано;
- без лишней бизнес-логики;
- с явными props.

Компонент должен обрабатывать:

- loading state;
- error state;
- empty state;
- disabled state;
- mobile layout, если он видимый пользователю.

---

## Правила маршрутов

Маршруты должны быть понятными.

Приватные маршруты должны проверять авторизацию.

Если пользователь не авторизован, он должен быть перенаправлен на login или получить понятное сообщение.

Нельзя показывать приватные данные до проверки доступа.

---

## Правила состояния

Состояние должно храниться на правильном уровне:

- локальное UI-состояние — в компоненте;
- состояние формы — в форме;
- пользователь/сессия — в auth/provider;
- серверные данные — через API/query layer;
- критические данные — источник истины backend/database.

Нельзя хранить критические данные только на клиенте.

---

## Правила API layer

API-запросы должны быть вынесены из UI-компонентов, если используются повторно или содержат логику.

Рекомендуемые зоны:

```text
src/services/
src/lib/
src/features/<feature>/api.ts
```

API layer должен:

- типизировать входные данные;
- типизировать ответ;
- обрабатывать ошибки;
- не раскрывать секреты;
- не выполнять критическую бизнес-логику на frontend.

---

## Backend архитектура

Backend отвечает за:

- авторизацию;
- проверку прав;
- работу с секретными ключами;
- платежи;
- webhook;
- начисление токенов;
- списание токенов;
- роли;
- лимиты;
- доступы;
- интеграции с внешними API;
- критические операции;
- аудит и логи.

Frontend только отправляет запросы и отображает результат.

## База данных

Для каждой важной сущности продумать:

- название таблицы;
- поля;
- типы полей;
- обязательность;
- значения по умолчанию;
- связи;
- индексы;
- уникальные ограничения;
- проверки;
- права чтения;
- права создания;
- права изменения;

Если приложение связано с пользователями, лимитами, токенами, оплатой или AI, рассмотреть таблицы:

- `profiles`;
- `user_settings`;
- `plans`;
- `subscriptions`;
- `payments`;
- `payment_events`;
- `token_balances`;
- `token_transactions`;
- `usage_logs`;
- `ai_requests`;
- `audit_logs`.

Создавать только то, что нужно текущему этапу.

## История операций

Для критических действий нужна история:

- начисление токенов;
- списание токенов;
- покупка;
- возврат;
- изменение роли;
- изменение лимита;
- админское действие;
- webhook-событие;
- AI-запрос.

История нужна для отладки, поддержки и безопасности.

---

## Идемпотентность

Критические операции должны быть идемпотентными.

Особенно:

- webhook платежей;
- начисление токенов;
- выдача доступа;
- обработка внешних событий;
- повторные запросы.

Повтор одной операции не должен приводить к двойному начислению или повреждению данных.

---

## Архитектура платежей

Правильная схема:

1. Frontend показывает тарифы.
2. Пользователь выбирает `plan_code`.
3. Backend получает `plan_code`.
4. Backend берет цену и состав тарифа из базы или безопасной конфигурации.
5. Backend создает платеж.
6. Платежная система возвращает redirect/confirmation URL.
7. Пользователь оплачивает.
8. Платежная система отправляет webhook.
9. Backend проверяет webhook.
10. Backend идемпотентно начисляет доступ/токены.
11. Frontend показывает обновленное состояние.

---

## Архитектура AI API

Правильная схема:

1. Frontend отправляет запрос пользователя.
2. Backend проверяет JWT/session.
3. Backend проверяет лимит/баланс.
4. Backend вызывает AI API.
5. Backend сохраняет запрос и ответ.
6. Backend списывает токены только при успешном ответе, если такая логика предусмотрена.
7. Frontend получает результат.

Запрещено вызывать AI API напрямую из браузера с секретным ключом.

---

## Масштабируемость

Код должен выдерживать развитие проекта.

Нельзя:

- дублировать бизнес-логику;
- размазывать правила по компонентам;
- хранить критические условия только в UI;
- создавать огромные файлы без необходимости;
- привязывать все к одному экрану;
- смешивать auth, billing, AI и UI в одном месте.

---

## Критерий хорошей архитектуры

Хорошая архитектура отвечает на вопросы:

- Где UI?
- Где бизнес-логика?
- Где API-запросы?
- Где типы?
- Где проверка прав?
- Где секреты?
- Где источник истины?
- Где история критических операций?
- Как проверить сценарий?
- Как добавить следующую функцию без переписывания всего проекта?
