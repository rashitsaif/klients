# 02_DEVELOPMENT_PROCESS.md

## Назначение

Этот файл задает обязательный процесс разработки для ИИ-агента.

Цель — не допустить хаотичных изменений, сломанной сборки, фейковой готовности и нестабильного приложения.

---

## Общий цикл работы

Каждая задача выполняется по циклу:

1. Понять задачу.
2. Изучить проект.
3. Определить риски.
4. Составить короткий план.
5. Выделить один безопасный этап.
6. Реализовать этап.
7. Проверить результат.
8. Исправить ошибки.
9. Дать отчет.

---

## Перед началом разработки

Перед любыми изменениями в существующем проекте изучить:

- структуру папок;
- README;
- package.json / pyproject.toml / requirements.txt / docker-compose.yml;
- конфиги сборки;
- env examples;
- роутинг;
- точки входа;
- API/services;
- hooks;
- components;
- types;
- tests;
- migrations;
- CI/CD, если есть.

Если проект новый, сначала спроектировать:

- MVP-цель;
- стек;
- экраны;
- сущности;
- маршруты;
- структуру папок;
- основные сценарии;
- риски.

---

## Технический план перед реализацией

Перед кодом всегда дать короткий план:

```text
План:
1. ...
2. ...
3. ...

Затрагиваемые зоны:
- ...

Риски:
- ...

Первый безопасный этап:
- ...
```

План должен быть практичным, а не абстрактным.

---

## Правило одного этапа

Один этап = одна понятная задача.

Примеры правильных этапов:

- создать каркас приложения;
- настроить роутинг;
- сверстать базовый экран;
- подключить авторизацию;
- защитить приватные маршруты;
- создать таблицы базы;
- добавить RLS;
- подключить одну backend-функцию;
- исправить одну категорию ошибок;
- добавить один пользовательский сценарий.

Примеры плохих этапов:

- “сделать весь SaaS”;
- “сразу авторизация + платежи + AI + админка”;
- “переписать весь frontend и backend”;
- “улучшить всё”.

---

## Что нельзя смешивать в одном этапе

Не смешивать:

- авторизацию;
- платежи;
- AI API;
- базу данных;
- админку;
- большой редизайн;
- рефакторинг всего проекта;
- миграции;
- деплой;
- сложную бизнес-логику.

Если пользователь просит всё сразу, разбить на этапы и начать с первого.

---

## Реализация

Во время реализации:

- менять только нужные файлы;
- следовать существующему стилю проекта;
- не удалять рабочую логику;
- не менять публичные API без причины;
- не менять пользовательские тексты без разрешения;
- не добавлять лишние зависимости без необходимости;
- не создавать дублирующие компоненты, если можно использовать существующие;
- не оставлять console.log в production-коде без причины;
- не оставлять временные файлы;
- не оставлять TODO вместо обязательной логики;
- не оставлять мертвый код.

---

## Работа с зависимостями

Добавлять новую зависимость только если:

- без нее задача существенно сложнее;
- зависимость популярна и поддерживается;
- она не дублирует уже установленный пакет;
- она не создает лишний риск.

После добавления зависимости проверить:

- package.json;
- lock-файл;
- сборку;
- импорты.

---

## Работа с TypeScript

Правила:

- не использовать грубый `any`, если можно описать тип;
- типы выносить в отдельные файлы при повторном использовании;
- props компонентов должны быть типизированы;
- API responses должны иметь типы;
- формы должны иметь типы данных;
- ошибки должны обрабатываться безопасно.

`any` допустим только временно и с объяснением причины.

---

## Работа с UI

Каждый интерактивный экран должен учитывать:

- loading;
- error;
- empty;
- success;
- disabled state;
- validation;
- mobile layout.

Кнопка не должна выглядеть рабочей, если действие недоступно.

Форма не должна отправляться с очевидно некорректными данными.

---

## Работа с backend/API

Backend/API должны:

- валидировать входные данные;
- проверять авторизацию;
- проверять права доступа;
- не доверять frontend;
- возвращать понятные ошибки;
- логировать критические операции;
- не раскрывать секреты;
- быть идемпотентными для критических операций.

---

## Проверки после реализации

После каждого этапа выполнить доступные проверки:

```bash
npm run build
npm run lint
npm run typecheck
npm test
```

Если команды нет, указать:

```text
Команда не найдена в package.json.
```

Если проверка невозможна, указать:

```text
Не удалось проверить из-за ...
```

Если проверка упала, нельзя завершать задачу как готовую. Нужно исправить или честно указать, что осталось.

---

## Ручная проверка

После изменения описать ручной сценарий:

1. Открыть ...
2. Нажать ...
3. Ввести ...
4. Проверить ...
5. Ожидаемый результат ...

---

## Работа с ошибками сборки

Если сборка падает:

1. Прочитать ошибку.
2. Найти файл и строку.
3. Исправить причину.
4. Повторить сборку.
5. Проверить, что нет новых ошибок.

Нельзя маскировать ошибку удалением важной логики.

---

## Работа с регрессиями

После изменений проверить связанные сценарии.

Например:

- изменил авторизацию — проверить вход, выход, приватный маршрут;
- изменил форму — проверить валидацию и отправку;
- изменил API — проверить успешный и ошибочный ответ;
- изменил платежи — проверить, что начисление не происходит на frontend.

---

## Финальное состояние этапа

Этап считается завершенным только если:

- код реализован;
- проект не сломан;
- проверки выполнены или честно указано, почему они невозможны;
- ошибки обработаны;
- старые функции не сломаны;
- отчет дан по шаблону.
