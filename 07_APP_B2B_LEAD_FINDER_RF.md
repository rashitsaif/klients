# Техническое задание: веб-приложение для поиска B2B-клиентов малого бизнеса

## 1. Название проекта

Рабочее название: **B2B Lead Finder AI**

Альтернативные названия:
- AI-лидогенератор B2B
- Поиск клиентов для B2B-услуг
- КлиентПоиск AI
- B2B Контакты + AI-продажи

## 2. Суть приложения

Веб-приложение помогает находить потенциальных клиентов среди малого бизнеса, собирать их публичные бизнес-контакты в личном кабинете, сегментировать лиды, выгружать контакты в файл и генерировать с помощью встроенного ИИ-агента коммерческое предложение и скрипт переписки под конкретную услугу.

Основной сценарий:

1. Пользователь заходит в личный кабинет.
2. Выбирает нишу малого бизнеса, регион и тип услуги, которую хочет предложить.
3. Приложение ищет и собирает доступные бизнес-контакты из разрешенных источников.
4. Контакты сохраняются в личном кабинете пользователя.
5. Пользователь фильтрует, сортирует, помечает и выгружает контакты.
6. Встроенный AI-агент генерирует коммерческое предложение и скрипт переписки под выбранный сегмент.
7. Пользователь использует выгрузку и материалы для ручной или CRM-обработки лидов.

## 3. Главная цель MVP

Создать рабочее приложение, которое позволяет пользователю:

- зарегистрироваться;
- настроить поиск B2B-контактов;
- запустить задачу сбора контактов из разрешенных источников;
- увидеть найденные контакты в личном кабинете;
- отфильтровать и выгрузить контакты в CSV/XLSX;
- сгенерировать коммерческое предложение и скрипт переписки через AI-агента;
- сохранить историю поисков и генераций.

MVP не должен сразу решать все задачи автоматического маркетинга. Первая версия должна стабильно закрывать главный сценарий: **поиск → сохранение → фильтрация → выгрузка → AI-коммерческое предложение**.

## 4. Целевая аудитория

Основные пользователи:

1. Владельцы B2B-услуг:
   - бухгалтерские сервисы;
   - юридические услуги;
   - услуги по взысканию задолженности;
   - разработчики сайтов и приложений;
   - маркетинговые агентства;
   - внедренцы CRM;
   - интеграторы AI-решений;
   - консалтинг для малого бизнеса.

2. Малые команды продаж:
   - менеджеры по продажам;
   - фрилансеры;
   - агентства;
   - предприниматели без отдела продаж.

3. Создатели SaaS и AI-продуктов:
   - ищут первых клиентов;
   - тестируют офферы;
   - собирают базы для ручного outreach.

## 5. Ключевая бизнес-ценность

Пользователь получает не просто список контактов, а готовый набор для начала продаж:

- база потенциальных клиентов;
- сегментация по нише и региону;
- понимание, что именно предложить;
- готовое коммерческое предложение;
- скрипт первого сообщения;
- варианты follow-up;
- выгрузка для CRM или ручной обработки.

## 6. Важное юридическое и этическое ограничение

Приложение должно проектироваться как инструмент для работы с разрешенными, публичными и законно полученными бизнес-данными.

Запрещено проектировать приложение как инструмент для:

- взлома сайтов;
- обхода CAPTCHA;
- обхода авторизации;
- обхода технических ограничений источников;
- массового спама;
- автоматической отправки рекламных сообщений без согласия;
- сбора персональных данных без правового основания;
- парсинга закрытых баз;
- использования украденных баз;
- нарушения robots.txt и условий использования источников;
- скрытого сбора данных.

Приложение должно:

- хранить источник каждого контакта;
- хранить дату сбора;
- хранить тип источника;
- позволять удалить контакт;
- иметь список исключений / suppression list;
- иметь пометку “не использовать для рассылки”;
- не отправлять рекламные сообщения автоматически в MVP;
- показывать пользователю предупреждение о необходимости соблюдать закон о персональных данных, рекламе и правила конкретных площадок.

## 7. Источники данных для MVP

### 7.1. Разрешенные источники первой версии

В MVP использовать только безопасные и контролируемые источники:

1. Ручной импорт пользователем:
   - CSV;
   - XLSX;
   - вставка списка сайтов;
   - вставка списка компаний.

2. Официальные открытые данные:
   - ЕГРЮЛ/ЕГРИП через официальные форматы или разрешенный доступ;
   - другие официальные открытые реестры, если их использование разрешено.

3. Публичные сайты компаний:
   - только если пользователь указывает список сайтов;
   - только открытые страницы контактов;
   - без обхода защит;
   - с rate limit;
   - с уважением robots.txt.

4. Поисковые API или business directory API:
   - только через официальные API;
   - только по правилам провайдера;
   - с хранением условий источника в настройках.

### 7.2. Источники не для MVP

В первой версии не реализовывать:

- массовый парсинг соцсетей;
- парсинг маркетплейсов с обходом ограничений;
- парсинг карт без официального API;
- обход CAPTCHA;
- авторизованный парсинг чужих кабинетов;
- автоматический сбор персональных email/телефонов физических лиц;
- автоматическую рекламную рассылку.

## 8. Режимы работы приложения

### 8.1. Режим “Импорт базы”

Пользователь загружает файл CSV/XLSX или вставляет список компаний/сайтов.

Приложение:

- валидирует файл;
- нормализует данные;
- удаляет дубли;
- сохраняет контакты в личном кабинете;
- показывает отчет импорта.

### 8.2. Режим “Поиск по нише”

Пользователь задает:

- регион;
- нишу;
- тип бизнеса;
- ключевые слова;
- желаемый размер бизнеса;
- источник;
- лимит результатов.

Приложение создает поисковую задачу.

### 8.3. Режим “Обогащение контактов”

Для уже найденной компании приложение пытается дополнить карточку:

- сайт;
- email компании;
- телефон компании;
- город;
- адрес;
- отрасль;
- описание деятельности;
- источник;
- ссылки на страницы контактов.

В MVP обогащение должно быть простым и безопасным: без обхода защит и без агрессивного парсинга.

### 8.4. Режим “AI-коммерческое предложение”

Пользователь выбирает:

- один лид;
- группу лидов;
- сегмент;
- услугу, которую предлагает;
- тон общения;
- цель сообщения.

AI-агент генерирует:

- краткое коммерческое предложение;
- первое сообщение;
- follow-up 1;
- follow-up 2;
- скрипт переписки;
- обработку типовых возражений;
- короткий оффер;
- тему письма, если нужен email-формат.

## 12. Права доступа и безопасность

### 12.1. Главный принцип

Пользователь видит и изменяет только свои данные.

### 12.2. Обязательные правила

- Включить RLS, если используется Supabase.
- Все таблицы пользовательских данных должны иметь user_id.
- Пользователь не может читать чужие проекты.
- Пользователь не может читать чужие лиды.
- Пользователь не может экспортировать чужие контакты.
- Пользователь не может видеть чужие AI-генерации.
- Пользователь не может изменять системные источники.
- Админские действия проверяются на backend.
- API-ключи внешних сервисов не попадают во frontend.
- AI API вызывается только через backend.
- Парсинг и фоновые задачи выполняются только на backend.

## 14. AI-агент внутри приложения

### 14.1. Назначение AI-агента

AI-агент помогает пользователю превращать найденные контакты в готовые материалы для продаж.

Он не должен автоматически рассылать сообщения в MVP.

Он генерирует тексты, которые пользователь проверяет и использует вручную.

### 14.2. Входные данные AI-агента

- описание услуги пользователя;
- ниша клиента;
- регион клиента;
- данные компании;
- боль клиента;
- желаемый тон;
- цель сообщения;
- формат материала;
- ограничения;
- преимущества услуги;
- CTA.

### 14.3. Типы генераций

1. Коммерческое предложение.
2. Первое сообщение.
3. Follow-up 1.
4. Follow-up 2.
5. Скрипт переписки.
6. Обработка возражений.
7. Короткий оффер.
8. Тема письма.
9. Сообщение в мессенджер.
10. Сообщение для звонка.

### 14.4. Правила AI-генерации

AI должен:

- писать конкретно под нишу клиента;
- не использовать слабые формулировки;
- не обещать незаконные или недостоверные результаты;
- не писать спамный текст;
- не имитировать личное знакомство, если его нет;
- не утверждать, что клиент уже давал согласие;
- не придумывать факты о компании;
- отделять факты от предположений;
- предлагать понятный CTA.

### 14.5. Пример структуры результата AI

Для коммерческого предложения:

- заголовок;
- короткое вступление;
- проблема клиента;
- решение;
- выгода;
- что входит;
- быстрый следующий шаг;
- CTA.

Для скрипта переписки:

- первое сообщение;
- если ответил “интересно”;
- если ответил “дорого”;
- если ответил “неактуально”;
- если не ответил;
- follow-up через 2 дня;
- follow-up через 5 дней.

## 16. Лид-скоринг

MVP-скоринг может быть простым.

Параметры:

- есть сайт;
- есть email;
- есть телефон;
- подходит регион;
- подходит ниша;
- есть признаки малого бизнеса;
- есть связь с услугой пользователя;
- нет признака do_not_contact.

Оценка:

- 0–30: слабый лид;
- 31–60: средний лид;
- 61–100: хороший лид.

Скоринг должен быть объяснимым: пользователь должен видеть, почему лид получил такую оценку.

3. Личный кабинет.
4. Проекты.
5. Импорт контактов.
6. Простая задача поиска по разрешенному источнику или списку сайтов пользователя.
7. Таблица лидов.
8. Карточка лида.

14. Базовая безопасность и RLS.
15. README и инструкция запуска.

## 18. Что не входит в MVP

В MVP не входит:

- автоматическая массовая рассылка;
- интеграция с email/SMS/мессенджерами для отправки;
- обход CAPTCHA;
- парсинг закрытых источников;
- обход ограничений сайтов;
- покупка или загрузка украденных баз;
- сложный ML-скоринг;
- автоматические платежи;
- подписки;
- биллинг;
- интеграция с CRM;
- ЭДО;
- глубокая аналитика продаж;
- мобильное приложение.

## 21. Критерии готовности MVP

MVP считается готовым, если:

- пользователь может зарегистрироваться;
- пользователь может создать проект;
- пользователь может импортировать или найти контакты из разрешенного источника;
- пользователь видит контакты в личном кабинете;
- пользователь может фильтровать контакты;
- пользователь может открыть карточку лида;
- пользователь может выгрузить контакты в CSV/XLSX;
- пользователь может сгенерировать КП и скрипт переписки;
- пользователь видит историю генераций;
- пользователь не видит чужие данные;
- секреты не попадают во frontend;
- есть README;
- приложение запускается и собирается.

## 22. Риски

### 22.1. Юридические риски

- персональные данные;
- рекламные рассылки без согласия;
- нарушение правил сайтов-источников;
- хранение контактных данных без понятного основания;
- жалобы на спам.

Меры снижения:

- не делать автоматическую рассылку в MVP;
- хранить источник контакта;
- хранить дату сбора;
- добавить suppression list;
- добавить предупреждение о законности использования;
- использовать только разрешенные источники;
- соблюдать условия источников;
- не собирать закрытые данные;
- не обходить техническую защиту.

### 22.2. Технические риски

- блокировки источников;
- дубли контактов;
- низкое качество данных;
- ошибки импорта;
- долгие фоновые задачи;
- перегрузка API;
- ошибки AI-генерации.

Меры снижения:

- rate limits;
- очередь задач;
- дедупликация;
- логирование;
- статусы задач;
- ручная проверка лидов;
- простые источники на MVP.

### 22.3. Продуктовые риски

- пользователь получит много нерелевантных контактов;
- пользователь будет ждать “волшебную базу”;
- AI может писать слабые тексты;
- без CRM-интеграции обработка может быть ручной.

Меры снижения:

- скоринг;
- фильтры;
- понятные ограничения MVP;
- хорошие шаблоны AI;
- экспорт для CRM.

## 23. Ограничения для ИИ-агента разработки

При реализации приложения агент обязан:

- работать по этапам;
- не делать массовый парсинг сразу;
- не реализовывать автоматическую рассылку в MVP;
- не добавлять обход CAPTCHA;
- не хранить секреты во frontend;
- не делать API-вызовы с секретами из браузера;
- не отключать RLS;
- не создавать фейковые “успешные” интеграции;
- не переходить к AI до готовой базы и лидов;
- не переходить к экспорту до готовой таблицы лидов;
- после каждого этапа запускать проверки.

## 25. Главная формула приложения

Приложение должно давать пользователю цепочку:

**ниша → поиск контактов → база лидов → фильтр → экспорт → AI-коммерческое предложение → скрипт переписки → ручная обработка продаж**

Главное: сначала стабильный и законный MVP, потом расширение источников, CRM, платежи, AI-скоринг и автоматизация.


## 26. Платежи и тарифы

Платежи не входят в самый первый технический MVP, но должны быть предусмотрены в архитектуре как отдельный этап после реализации авторизации, базы данных, личного кабинета, лидов, экспорта и AI-генератора.

### 26.1. Цель платежей

Монетизация приложения через платный доступ к лимитам:

* количество найденных/импортированных контактов;
* количество экспортов;
* количество AI-генераций;
* количество проектов;
* доступ к расширенным функциям.

### 26.2. Тарифы MVP

Рекомендуемые тарифы:

#### Free

* 1 проект;
* до 50 контактов;
* до 3 экспортов;
* до 5 AI-генераций;
* без массового поиска;
* только тестовый режим.

#### Start

* до 1 000 контактов в месяц;
* до 20 экспортов;
* до 50 AI-генераций;
* до 3 проектов.

#### Pro

* до 10 000 контактов в месяц;
* до 200 экспортов;
* до 500 AI-генераций;
* до 20 проектов;
* расширенные фильтры;
* история генераций.

#### Business

* повышенные лимиты;
* командный доступ;
* админ-панель;
* приоритетные задачи;
* индивидуальные лимиты.

### 26.3. Платежный провайдер

Для российского рынка использовать:

* ЮKassa как основной вариант;
* CloudPayments или Т-Банк как альтернативу.

Платежи должны подключаться только через backend.

### 26.4. Архитектура платежей

Правильная схема:

1. Пользователь выбирает тариф.
2. Frontend отправляет только `plan_code`.
3. Backend берет цену, лимиты и параметры тарифа из таблицы `plans`.
4. Backend создает платеж у платежного провайдера.
5. Пользователь переходит на страницу оплаты.
6. Платежный провайдер отправляет webhook.
7. Backend проверяет webhook.
8. Backend проверяет идемпотентность.
9. Backend активирует тариф или начисляет лимиты.
10. Frontend только показывает обновленный статус.

### 26.5. Запрещено

* передавать сумму с frontend как источник истины;
* подтверждать оплату на frontend;
* активировать тариф на success page;
* менять статус платежа из браузера;
* хранить secret key во frontend;
* начислять лимиты без webhook;
* допускать двойное начисление при повторном webhook.

5. лидов;
6. импорта;
