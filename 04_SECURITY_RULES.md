# 04_SECURITY_RULES.md

## Назначение

Этот файл задает обязательные правила безопасности для ИИ-агента.

Главный принцип:

> Frontend нельзя доверять. Все критические действия выполняются только на backend.

---

## Базовые правила безопасности

Всегда проверять:

1. Пользователь авторизован?
2. Имеет ли пользователь право на действие?
3. Может ли пользователь прочитать чужие данные?
4. Может ли пользователь изменить чужие данные?
5. Может ли пользователь подделать оплату?
6. Может ли пользователь сам себе начислить баланс, токены или доступ?
7. Не попадают ли секреты во frontend?
8. Не может ли критическая операция выполниться дважды?
9. Логируются ли важные действия?
10. Есть ли безопасная обработка ошибок?

---

## Что считается критической операцией

Критические операции:

- регистрация и вход;
- изменение email/пароля;
- изменение роли;
- выдача доступа;
- начисление токенов;
- списание токенов;
- изменение баланса;
- создание платежа;
- подтверждение платежа;
- обработка webhook;
- изменение подписки;
- вызов платного AI API;
- админские действия;
- удаление данных;
- экспорт данных.

Критические операции нельзя доверять frontend.

---

## Секреты

Запрещено хранить во frontend:

- API keys;
- secret keys;
- service role keys;
- webhook secrets;
- payment provider secret keys;
- private tokens;
- database credentials;
- admin keys.

Разрешено во frontend только:

- публичные anon keys, если это предусмотрено платформой;
- публичные URL;
- публичные конфиги, которые не дают доступа к приватным данным.

---

## Авторизация

Любой приватный сценарий должен проверять:

- наличие сессии;
- срок действия сессии;
- права пользователя;
- принадлежность данных пользователю.

Нельзя показывать приватные данные до завершения проверки.

---

## Правила доступа к данным

Пользователь может читать и изменять только свои данные, если бизнес-логика не предусматривает иное.

Для каждой сущности определить:

- кто может читать;
- кто может создавать;
- кто может изменять;
- кто может удалять;
- какие поля нельзя менять с клиента.

---

## RLS / Row Level Security

Если используется Supabase/PostgreSQL с RLS:

- RLS должен быть включен для пользовательских таблиц;
- политики должны ограничивать доступ по `auth.uid()`;
- пользователь не должен видеть чужие записи;
- пользователь не должен менять чужие записи;
- service role key нельзя использовать в браузере;
- обход RLS допустим только на backend для строго контролируемых операций.

Запрещено отключать RLS ради удобства.

---

## Платежи

Frontend не является источником истины для платежей.

Запрещено:

- подтверждать платеж на frontend;
- начислять доступ на success page;
- менять `payments.status` с клиента;
- передавать сумму с frontend как источник истины;
- доверять plan name/price с frontend;
- начислять токены без webhook;
- обрабатывать webhook без проверки подлинности;
- начислять дважды при повторном webhook.

Правильная схема:

1. Frontend отправляет только `plan_code` или ID тарифа.
2. Backend берет цену и состав тарифа из базы/конфига.
3. Backend создает платеж.
4. Платеж получает статус `pending`.
5. Платежная система отправляет webhook.
6. Backend проверяет подлинность webhook.
7. Backend проверяет статус платежа.
8. Backend проверяет, не обработано ли событие ранее.
9. Backend начисляет доступ/токены/баланс.
10. Backend пишет запись в историю операций.
11. Frontend только показывает результат.

---

## Идемпотентность платежей

Webhook может прийти повторно.

Обязательно хранить:

- payment id;
- provider event id, если есть;
- статус обработки;
- дату обработки;
- результат обработки.

Повторный webhook не должен повторно начислять доступ.

---

## AI API

AI API вызываются только через backend.

Запрещено:

- вызывать OpenAI/DeepSeek/Anthropic напрямую из браузера с secret key;
- передавать secret key клиенту;
- списывать токены на frontend;
- проверять лимиты только на frontend;
- не логировать платные AI-запросы;
- показывать пользователю внутренние ошибки провайдера с секретными деталями.

Правильная схема:

1. Frontend отправляет запрос.
2. Backend проверяет пользователя.
3. Backend проверяет лимит/баланс.
4. Backend вызывает AI API.
5. Backend сохраняет запрос/ответ или метаданные.
6. Backend списывает токены по правилам.
7. Frontend показывает результат.

---

## Валидация данных

Все входные данные с клиента считаются недоверенными.

Backend должен валидировать:

- обязательные поля;
- типы;
- длину строк;
- допустимые значения;
- числовые диапазоны;
- принадлежность ID пользователю;
- права на действие.

Frontend-валидация нужна для UX, но она не заменяет backend-валидацию.

---

## Ошибки

Ошибки должны быть безопасными.

Нельзя раскрывать пользователю:

- secret keys;
- stack traces в production;
- внутренние SQL-запросы;
- приватные данные других пользователей;
- подробности инфраструктуры, если это создает риск.

Пользователь должен получать понятное сообщение.

Внутренние детали должны логироваться безопасно.

---

## Админские действия

Админские операции должны:

- проверять роль;
- выполняться только на backend;
- логироваться;
- иметь историю;
- быть ограничены минимально необходимыми правами.

Нельзя делать админку, которая работает только через скрытую кнопку на frontend без backend-проверки роли.

---

## Файлы и загрузки

Если есть загрузка файлов:

- проверять тип;
- проверять размер;
- ограничивать доступ;
- хранить приватные файлы в приватном storage;
- не позволять пользователю читать чужие файлы;
- не доверять имени файла;
- не исполнять загруженный файл.

---

## Минимальный security-чек перед завершением

Перед отчетом проверить:

- нет ли секретов во frontend;
- нет ли service role key в клиентском коде;
- не ослаблена ли авторизация;
- не отключен ли RLS;
- не может ли пользователь менять чужие данные;
- не может ли пользователь сам себе выдать доступ;
- не подтверждается ли платеж без webhook;
- обрабатываются ли ошибки безопасно.

---

## Критическая ошибка

Следующие ситуации считаются критическими ошибками:

- secret key во frontend;
- service role key во frontend;
- платеж подтверждается на success page;
- токены начисляются на клиенте;
- пользователь может изменить чужие данные;
- RLS отключен без безопасной замены;
- админские права проверяются только в UI;
- AI API key доступен в браузере.

При обнаружении критической ошибки сначала исправить ее, потом продолжать работу.
