# 05_QA_CHECKLIST.md

## Назначение

Этот файл задает обязательный чеклист проверки после каждого этапа разработки.

Главное правило:

> Если результат не проверен, нельзя выдавать его как полностью готовый.

---

## 1. Проверка запуска и сборки

Проверить доступные команды:

```bash
npm install
npm run build
npm run lint
npm run typecheck
npm test
```

Выполнять только те команды, которые применимы к проекту.

Если команда отсутствует, указать:

```text
Команда не найдена.
```

Если команда не запускалась, указать причину.

Если команда упала, указать ошибку и исправить, если возможно.

---

## 2. Проверка структуры проекта

Проверить:

- файлы лежат в правильных папках;
- нет случайных временных файлов;
- нет дублей компонентов без необходимости;
- нет хаотичного смешивания UI и бизнес-логики;
- нет огромных файлов, которые можно разумно разделить;
- нет случайно удаленных рабочих файлов;
- нет лишних зависимостей.

---

## 3. Проверка TypeScript/типов

Проверить:

- нет критических TypeScript-ошибок;
- props компонентов типизированы;
- API responses типизированы;
- формы типизированы;
- нет грубого `any` без причины;
- нет несовместимых типов;
- нет сломанных импортов;
- нет неиспользуемых импортов, если линтер это проверяет.

---

## 4. Проверка UI

Проверить:

- экран открывается;
- кнопки работают;
- формы отправляются;
- есть loading state;
- есть error state;
- есть empty state;
- есть disabled state при недоступном действии;
- сообщения понятны пользователю;
- интерфейс не ломается на мобильной ширине;
- важные элементы не перекрываются;
- нет нерабочих CTA, если они должны работать.

---

## 5. Проверка форм

Для каждой формы проверить:

- обязательные поля;
- некорректные значения;
- пустую отправку;
- успешную отправку;
- ошибку API;
- повторную отправку;
- disabled/loading во время отправки;
- понятное сообщение об ошибке.

---

## 6. Проверка авторизации

Если задача затрагивает auth, проверить:

- регистрация;
- вход;
- выход;
- восстановление доступа, если есть;
- приватный маршрут без сессии;
- приватный маршрут с сессией;
- редирект неавторизованного пользователя;
- пользователь не видит чужие данные.

---

## 7. Проверка данных

Проверить:

- пользователь видит правильные данные;
- пользователь не видит чужие данные;
- создание записи работает;
- изменение записи работает;
- удаление работает, если предусмотрено;
- ошибки API обрабатываются;
- пустой список отображается нормально;
- повторные запросы не ломают состояние.

---

## 8. Проверка безопасности

Проверить:

- нет API secret keys во frontend;
- нет service role key во frontend;
- критические операции не выполняются на клиенте;
- backend проверяет авторизацию;
- backend проверяет права;
- пользователь не может изменить чужие данные;
- ошибки не раскрывают секреты;
- RLS не отключен без безопасной причины.

---

## 9. Проверка платежей

Если задача затрагивает платежи, проверить:

- frontend отправляет только безопасный идентификатор тарифа;
- backend берет цену из базы/конфига;
- платеж создается на backend;
- статус платежа не меняется с клиента;
- success page ничего не начисляет;
- webhook проверяется;
- webhook идемпотентен;
- повторный webhook не начисляет доступ дважды;
- операция записывается в историю;
- пользователь видит обновленный статус после backend-подтверждения.

---

## 10. Проверка AI API

Если задача затрагивает AI API, проверить:

- AI API key не попадает во frontend;
- запрос идет через backend;
- backend проверяет пользователя;
- backend проверяет лимит/баланс;
- backend обрабатывает ошибку AI-провайдера;
- списание происходит по правилам;
- запрос логируется, если это требуется;
- frontend показывает loading/error/result.

---

## 11. Проверка backend/API

Проверить:

- входные данные валидируются;
- ошибки возвращаются в понятном формате;
- авторизация проверяется;
- права проверяются;
- критические операции логируются;
- повторные запросы не создают дубли, если это важно;
- нет утечки секретов в ответах.

---

## 12. Проверка базы данных

Если задача затрагивает базу, проверить:

- таблицы созданы корректно;
- поля имеют правильные типы;
- обязательные поля отмечены;
- индексы добавлены там, где нужно;
- связи корректны;
- уникальные ограничения есть там, где нужны;
- RLS включен, если применяется;
- политики доступа соответствуют логике.

---

## 13. Проверка регрессий

Проверить связанные сценарии.

Примеры:

- изменили auth — проверить приватные маршруты;
- изменили форму — проверить успешную и ошибочную отправку;
- изменили API — проверить UI, который его использует;
- изменили типы — проверить сборку;
- изменили платежи — проверить, что начисления не на frontend;
- изменили routing — проверить навигацию.

---

## 14. Минимальный ручной сценарий

В отчете указать:

```text
Как проверить вручную:
1. ...
2. ...
3. ...
Ожидаемый результат: ...
```

---

## 15. Критерии готовности

Функция считается готовой, если:

- код реализован;
- проект собирается;
- критических ошибок нет;
- основной сценарий работает;
- ошибки обработаны;
- безопасность не ослаблена;
- старые функции не сломаны;
- отчет содержит проверки.

---

## 16. Критерии неготовности

Функция не считается готовой, если:

- сборка падает;
- есть критические TypeScript-ошибки;
- приложение не запускается;
- есть сломанные импорты;
- нет обработки ошибок;
- секреты попали во frontend;
- платеж подтверждается без webhook;
- пользователь может изменить чужие данные;
- нет проверки авторизации;
- проверка не выполнена и это скрыто.

---

## 17. Формулировка при неполной проверке

Если не удалось проверить всё, писать честно:

```text
Реализация выполнена, но полная проверка не завершена.

Не удалось проверить:
- ...

Причина:
- ...

Что нужно сделать для проверки:
- ...
```

Нельзя писать “всё готово”, если это не подтверждено.
