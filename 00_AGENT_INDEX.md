# 00_AGENT_INDEX.md

## Назначение

Это главный навигационный файл для ИИ-агента, который помогает создавать стабильные функциональные приложения.

Агент обязан использовать этот файл перед любой задачей по разработке, архитектуре, исправлению ошибок, ревью кода, доработке интерфейса, подключению backend, интеграций, платежей, AI API, базы данных, авторизации и деплоя.

Главный принцип:

> Сначала определить тип задачи и нужные инструкции. Потом составить план. Потом выполнить один безопасный этап. Потом проверить результат.

---

## Обязательные файлы инструкций

В проекте должны быть доступны следующие файлы:

- `00_AGENT_INDEX.md` — карта инструкций.
- `01_CORE_BEHAVIOR.md` — базовое поведение агента.
- `02_DEVELOPMENT_PROCESS.md` — процесс разработки по этапам.
- `03_ARCHITECTURE_RULES.md` — архитектура приложений.
- `04_SECURITY_RULES.md` — безопасность.
- `05_QA_CHECKLIST.md` — проверки качества.
- `06_TASK_REPORT_TEMPLATE.md` — шаблон итогового отчета.

### Любая задача по разработке

Использовать:

- `01_CORE_BEHAVIOR.md`
- `02_DEVELOPMENT_PROCESS.md`
- `05_QA_CHECKLIST.md`
- `06_TASK_REPORT_TEMPLATE.md`

### Создание приложения с нуля

Использовать:

- `01_CORE_BEHAVIOR.md`
- `02_DEVELOPMENT_PROCESS.md`
- `03_ARCHITECTURE_RULES.md`
- `04_SECURITY_RULES.md`
- `05_QA_CHECKLIST.md`
- `06_TASK_REPORT_TEMPLATE.md`

### Доработка существующего проекта

Использовать:

- `01_CORE_BEHAVIOR.md`
- `02_DEVELOPMENT_PROCESS.md`
- `03_ARCHITECTURE_RULES.md`
- `05_QA_CHECKLIST.md`
- `06_TASK_REPORT_TEMPLATE.md`

Перед изменениями обязательно изучить:

- точки входа;
- package/config файлы;
- существующие паттерны;
- текущие ошибки, если они есть.

### Исправление ошибок

Использовать:

- `01_CORE_BEHAVIOR.md`
- `02_DEVELOPMENT_PROCESS.md`
- `05_QA_CHECKLIST.md`
- `06_TASK_REPORT_TEMPLATE.md`

Правило:

> Сначала воспроизвести или логически локализовать ошибку. Потом исправлять минимальным изменением. Потом проверить, что ошибка ушла и не появилась регрессия.

### Рефакторинг

Использовать:

- `01_CORE_BEHAVIOR.md`
- `02_DEVELOPMENT_PROCESS.md`
- `03_ARCHITECTURE_RULES.md`
- `05_QA_CHECKLIST.md`
- `06_TASK_REPORT_TEMPLATE.md`

Правило:

> Рефакторинг не должен менять поведение приложения, если пользователь явно не попросил изменить поведение.

### Архитектура, структура проекта, план MVP

Использовать:

- `01_CORE_BEHAVIOR.md`
- `02_DEVELOPMENT_PROCESS.md`
- `03_ARCHITECTURE_RULES.md`
- `04_SECURITY_RULES.md`

### Авторизация, роли, личный кабинет, приватные маршруты

Использовать:

- `03_ARCHITECTURE_RULES.md`
- `04_SECURITY_RULES.md`
- `05_QA_CHECKLIST.md`

Правило:

> Приватные данные доступны только авторизованному пользователю и только в рамках его прав.

### База данных, модели, таблицы, миграции

Использовать:

- `03_ARCHITECTURE_RULES.md`
- `04_SECURITY_RULES.md`
- `05_QA_CHECKLIST.md`

Правило:

> Для каждой важной сущности должны быть продуманы поля, связи, индексы, ограничения, права доступа и история критических операций.

### Supabase, RLS, Edge Functions

Использовать:

- `03_ARCHITECTURE_RULES.md`
- `04_SECURITY_RULES.md`
- `05_QA_CHECKLIST.md`

Правило:

> RLS нельзя отключать ради удобства. Service role key никогда не должен попадать во frontend.

### Платежи, подписки, токены, баланс, тарифы, доступы

Использовать:

- `03_ARCHITECTURE_RULES.md`
- `04_SECURITY_RULES.md`
- `05_QA_CHECKLIST.md`
- `06_TASK_REPORT_TEMPLATE.md`

Правило:

> Frontend не подтверждает оплату, не начисляет баланс, не выдает доступы и не меняет статус платежа. Источник истины — backend + webhook платежной системы.

### AI API, OpenAI, DeepSeek, Anthropic, генерации

Использовать:

- `03_ARCHITECTURE_RULES.md`
- `04_SECURITY_RULES.md`
- `05_QA_CHECKLIST.md`

Правило:

> AI API вызываются только через backend. API-ключи не попадают во frontend. Лимиты и списания проверяются на backend.

### UI, адаптивность, компоненты, формы

Использовать:

- `01_CORE_BEHAVIOR.md`
- `02_DEVELOPMENT_PROCESS.md`
- `03_ARCHITECTURE_RULES.md`
- `05_QA_CHECKLIST.md`

Правило:

> Красивый интерфейс не должен ломать функциональность. Формы должны иметь валидацию, loading/error/empty/success состояния и понятное поведение.

### Финальная стабилизация

Использовать:

- `01_CORE_BEHAVIOR.md`
- `02_DEVELOPMENT_PROCESS.md`
- `04_SECURITY_RULES.md`
- `05_QA_CHECKLIST.md`
- `06_TASK_REPORT_TEMPLATE.md`

## Обязательный порядок работы

Для любой задачи агент обязан:

1. Определить тип задачи.
2. Выбрать нужные инструкции по этому файлу.
3. Кратко сформулировать технический план.
4. Определить минимальный безопасный этап.
5. Реализовать только этот этап.
6. Проверить результат.
7. Исправить найденные ошибки.
8. Дать отчет по `06_TASK_REPORT_TEMPLATE.md`.

---

## Если пользователь просит сделать большое приложение

Не делать всё сразу.

Сначала выдать:

1. MVP-цель.
2. Список экранов.
3. Основные сущности.
4. Архитектуру.
5. Этапы реализации.
6. Риски.
7. Первый безопасный этап.

Потом реализовывать по одному этапу.

---

## Если пользователь просит “сделай полностью”

Разрешено сделать полный результат только если задача маленькая и проверяемая.

Если задача большая, агент обязан разделить ее на этапы и начать с первого рабочего этапа.

---

## Главный критерий

Задача не считается выполненной, если:

- приложение не запускается;
- сборка падает;
- есть критические ошибки типов;
- есть сломанные импорты;
- секреты попали во frontend;
- платежи подтверждаются без webhook;
- пользователь может изменить чужие данные;
- нет понятного отчета о проверках.
